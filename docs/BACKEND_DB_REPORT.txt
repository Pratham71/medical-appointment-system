MEDICAL APPOINTMENT SYSTEM - BACKEND AND DATABASE REPORT

1. PROJECT OVERVIEW

The Medical Appointment System is a DBMS-focused application built for a college infirmary. The backend provides APIs for authentication, appointment booking, doctor workflows, student/patient records, staff walk-in handling, admin management, emergency alerts, reports, prescriptions, and medical certificates.

The backend is implemented using FastAPI and MySQL. The system follows a strict layered architecture:

routes -> services -> repositories -> queries -> database

This structure keeps the API code clean and separates responsibilities clearly:

- Routes define HTTP endpoints and request/response handling.
- Services contain business rules and validation.
- Repositories manage database transactions and coordinate query calls.
- Query modules contain all raw SQL statements.
- Database files define schema, seed data, migrations, views, indexes, triggers, and connection handling.

The application uses MySQL as the selected database provider for the MVP. The schema is normalized up to 3NF for the current project scope. Lookup tables are used for roles, appointment statuses, slot statuses, and certificate types. Repeating groups are separated into independent tables, such as prescriptions and prescription_items.


2. BACKEND TECHNOLOGY STACK

- Python: Backend programming language.
- FastAPI: API framework used to build REST endpoints.
- Pydantic: Request and response schema validation.
- MySQL: Main database provider.
- mysql-connector-python: MySQL connection and connection pooling.
- bcrypt: Password hashing.
- python-jose: JWT token creation and validation.
- uv: Python dependency and environment manager.
- pytest: Automated backend and database test runner.
- ruff: Python linting and style checking.


3. HIGH-LEVEL BACKEND ARCHITECTURE

The backend code is inside:

app/backend/app/

Main flow of a request:

1. The frontend or API client sends a request to a FastAPI endpoint.
2. The route validates path/query/body data using Pydantic schemas.
3. Authentication and role checks run through FastAPI dependencies.
4. The route calls the relevant service function.
5. The service applies business rules, status checks, permission rules, and validation.
6. The service calls the repository layer.
7. The repository opens a database connection or transaction.
8. The repository calls query functions.
9. Query functions execute parameterized raw SQL against MySQL.
10. The result is converted into schema objects and returned as JSON.

app/backend/app/api/dependencies.py

- Contains reusable FastAPI dependencies.
- Extracts authenticated users from JWT bearer tokens.
- Enforces role-based access control.
- Resolves the current student's student_id from the logged-in user.
- Resolves the current doctor's staff_id from the logged-in user.
- Checks appointment access permissions.
- Checks student medical record access permissions.

Important functions:

- get_current_user()
  - Reads Authorization: Bearer token.
  - Validates token.
  - Loads current user from database.

- require_roles()
  - Restricts an endpoint to one or more roles.
  - Used for admin-only, doctor-only, staff-only, and patient routes.

- require_student_id()
  - Allows student, professor, college-staff, and hostel-staff roles.
  - Returns the linked student_id.
  - These roles share the patient workflow.

- require_doctor_staff_id()
  - Allows only doctor role.
  - Returns the linked staff_id.

- ensure_appointment_access()
  - Ensures a user can access a specific appointment.
  - Supports patient, doctor, admin, and staff permission checks.

- ensure_student_record_access()
  - Ensures access to a student's medical records.
  - Students can access their own records.
  - Admin can access all records.
  - Doctors can access records for patients they have seen.

6. API ROUTES

Folder:

app/backend/app/api/routes/

Purpose:

- Contains all FastAPI endpoint definitions.
- Routes stay thin and call services for business logic.
- Routes do not contain raw SQL.

app/backend/app/api/routes/__init__.py

- Package marker file.

app/backend/app/api/routes/auth.py

Purpose:

- Handles authentication, signup, login, logout, and current-user lookup.

Endpoints:

- POST /auth/signup
  - Creates a new user account.
  - By default, signup creates a student/patient account.
  - Creates a linked student profile.
  - Returns a JWT access token.

- POST /auth/login
  - Authenticates email and password.
  - Verifies bcrypt password hash.
  - Blocks repeated failed login attempts.
  - Returns JWT access token and user information.

- POST /auth/logout
  - Stateless logout endpoint.
  - Confirms logout on the backend side.
  - Since JWT is stateless, frontend removes the token.

- GET /auth/me
  - Returns the currently authenticated user.
  - Uses the bearer token.

app/backend/app/api/routes/appointments.py

Purpose:

- Handles doctor listing, slot listing, booking, cancellation, and completion.

Endpoints:

- GET /appointments/doctors?for_date=YYYY-MM-DD
  - Returns all doctors and availability status for the chosen date.
  - Includes doctors even if unavailable.
  - Uses weekly availability and date overrides.

- GET /appointments/slots?from_date=YYYY-MM-DD
  - Returns available appointment slots for a selected date.
  - Filters out unavailable, booked, and elapsed same-day slots.

- GET /appointments/slots/all?doctor_id=ID&slot_date=YYYY-MM-DD
  - Returns all slots for one doctor on one date.
  - Includes available, booked, and unavailable slot statuses.

- POST /appointments/book
  - Books a slot for the authenticated patient.
  - Uses authenticated student_id, not caller-supplied student_id.
  - Requires idempotency key for replay safety.
  - Prevents double booking.

- PATCH /appointments/{appointment_id}/cancel
  - Cancels an appointment.
  - Patients can cancel their own appointment.
  - Doctors, staff, and admins must provide a cancellation reason.
  - Cancelling releases the slot.

- PATCH /appointments/{appointment_id}/complete
  - Marks an appointment as completed.
  - Only doctors and admins can complete appointments.
  - Cancelled appointments cannot be completed.

app/backend/app/api/routes/students.py

Purpose:

- Provides patient-facing data for student/professor/college-staff/hostel-staff workflows.

Endpoints:

- GET /students/dashboard
  - Returns counts and next appointment for the authenticated patient.

- GET /students/appointments
  - Returns all appointments for the authenticated patient.
  - Includes booked, completed, and cancelled appointments.

- GET /students/reports
  - Returns medical report summaries for the authenticated patient.

- GET /students/certificates
  - Returns certificate summaries for the authenticated patient.

- GET /students/emergency-alerts
  - Returns emergency alerts submitted by the authenticated patient.
  - Shows lifecycle status such as unread, acknowledged, or resolved.

app/backend/app/api/routes/doctors.py

Purpose:

- Provides doctor dashboard, appointments, availability management, appointment details, and patient history access.

Endpoints:

- GET /doctors/dashboard
  - Returns appointment and patient statistics for the authenticated doctor.

- GET /doctors/appointments
  - Returns all appointments assigned to the authenticated doctor.

- GET /doctors/availability
  - Returns weekly availability rules and date-specific overrides.

- PUT /doctors/availability/weekly/{weekday}
  - Updates recurring availability for a weekday.
  - Weekday values: 0 = Monday, 6 = Sunday.
  - By default, doctors are available Monday to Saturday and unavailable Sunday.

- PUT /doctors/availability/overrides/{override_date}
  - Creates or updates availability for a specific date.
  - Can mark doctor unavailable for one date.
  - If booked appointments exist on that date, the backend auto-cancels them.

- DELETE /doctors/availability/overrides/{override_date}
  - Removes a date-specific override.
  - Falls back to weekly availability.

- GET /doctors/patients/search?q=
  - Searches patients by name or roll number.
  - Doctors can only search patients they have seen.
  - Admins can search all patients.

- GET /doctors/appointment/{appointment_id}
  - Returns detailed appointment information.
  - Includes student, doctor, date, status, reason, notes, certificate info, and prescription context.

- GET /doctors/patient-history/{student_id}
  - Returns a patient's appointment history.
  - Doctors must have treated the patient before.
  - Admins can access all patient histories.

app/backend/app/api/routes/reports.py

Purpose:

- Handles medical notes, prescriptions, and full report viewing.

Endpoints:

- POST /reports/{appointment_id}/notes
  - Doctor/admin creates or updates medical notes.
  - Stores diagnosis and remarks.
  - Locked after appointment is completed or cancelled.

- POST /reports/{appointment_id}/prescription
  - Doctor/admin creates or replaces prescription items.
  - Prescriptions are stored separately from prescription items.
  - Locked after appointment is completed or cancelled.

- GET /reports/{appointment_id}
  - Returns a full report for one appointment.
  - Accessible to the patient, treating doctor, or admin.

app/backend/app/api/routes/certificates.py

Purpose:

- Handles medical certificate creation and lookup.

Endpoints:

- POST /certificates/{appointment_id}
  - Doctor/admin issues or updates a certificate.
  - Supports certificate type, issue date, leave start/end dates, and certificate notes.
  - Enforces appointment date rules.
  - Locked after appointment is completed or cancelled.

- GET /certificates/student/{student_id}
  - Returns all certificates for one student.
  - Accessible to the student, treating doctor, or admin.

app/backend/app/api/routes/emergencies.py

Purpose:

- Allows patients to create emergency alerts.

Endpoint:

- POST /emergency/alert
  - Creates an emergency alert for the authenticated patient.
  - Stores reason, location, contact number, message, and timestamp.

app/backend/app/api/routes/admin.py

Purpose:

- Provides admin-only system management APIs.
- Also allows admin/staff/doctor responders to manage emergency alert lifecycle.

Endpoints:

- GET /admin/dashboard
  - Returns system-wide counts for users, appointments, doctors, students, staff, and emergency alerts.

- GET /admin/users
  - Lists users.
  - Supports search, role filter, and limit.

- PATCH /admin/users/{user_id}/role
  - Allows admin to assign roles.
  - Roles include student, professor, college-staff, hostel-staff, doctor, staff, and admin.
  - Updates linked profile depending on role.

- PATCH /admin/users/{user_id}/deactivate
  - Soft-deactivates a user.
  - Does not hard-delete medical history.

- PATCH /admin/users/{user_id}/activate
  - Reactivates a deactivated user.

- DELETE /admin/users/{user_id}
  - Alias for soft-deactivation.
  - Keeps medical history intact.

- GET /admin/appointments
  - Lists appointments with filters.
  - Supports status, date range, doctor_id, student_id, and limit.

- GET /admin/students
  - Lists patient profiles.

- GET /admin/doctors
  - Lists doctor profiles.

- GET /admin/staff
  - Lists non-doctor staff profiles.

- GET /admin/emergency-alerts
  - Lists emergency alerts.
  - Accessible by admin, staff, and doctor.

- PATCH /admin/emergency-alerts/{alert_id}/acknowledge
  - Marks emergency alert as acknowledged.

- PATCH /admin/emergency-alerts/{alert_id}/resolve
  - Marks emergency alert as resolved with optional resolution note.

app/backend/app/api/routes/staff.py

Purpose:

- Provides front-desk/staff workflow APIs.

Endpoints:

- GET /staff/dashboard
  - Returns staff dashboard counts.

- GET /staff/appointments
  - Lists appointments for staff review.
  - Supports status and date range filters.

- GET /staff/patients/search?q=
  - Searches existing patient profiles.
  - Used before booking walk-ins.

- GET /staff/walk-ins
  - Lists staff-created walk-in appointments.

- POST /staff/walk-ins/book
  - Books an existing patient into an available appointment slot as a walk-in.

  - Creates bcrypt hash for a password.

  - Creates JWT with user_id, role, and expiration.

  - Validates JWT and returns token payload.

app/backend/app/core/security_controls.py

Purpose:

- Adds security controls around requests.

Classes:

- FixedWindowRateLimiter
  - Limits requests per client per time window.

- LoginAttemptTracker
  - Tracks failed login attempts.
  - Locks out email after too many failures.

- IdempotencyStore
  - Stores successful write responses by idempotency key.
  - Replays same response for duplicate requests with same body.
  - Rejects same idempotency key with different body.

Main middleware:

- security_request_middleware()
  - Applies rate limiting.
  - Requires Idempotency-Key for authenticated mutating requests.
  - Caches successful write responses under idempotency keys.

Write requests protected by idempotency:

- POST
- PUT
- PATCH
- DELETE

Auth login/logout are excluded from idempotency requirement.

16. DATABASE NORMALIZATION

The database is normalized up to 3NF for the MVP.

1NF:

- Tables use atomic fields.
- Repeating medicine items are separated into prescription_items.
- Repeating certificate types are separated into certificate_types.

2NF:

- Non-key fields depend on whole primary keys.
- Lookup values are separated from transactional records.

3NF:

- User role names are stored in roles, not repeated in users.
- Appointment statuses are stored in appointment_statuses.
- Slot statuses are stored in slot_statuses.
- Certificate type labels are stored in certificate_types.
- Appointment records do not duplicate doctor/student names; they reference slot and student records.
- Prescription items depend on prescription, not directly on appointment.

Practical note:

- Professor, college-staff, and hostel-staff currently use the students table as a shared patient profile table. This is acceptable for MVP, but the table could later be renamed to patient_profiles for clearer naming.


17. IMPORTANT DBMS CONCEPTS USED

Primary keys:

- Every main table has a primary key.

Foreign keys:

- Used to enforce relationships between users, roles, students, staff, appointments, slots, reports, prescriptions, certificates, and emergency alerts.

Unique constraints:

- Prevent duplicate emails, role names, roll numbers, employee numbers, status names, slot time ranges, and active bookings.

Check constraints:

- Validate student year level.
- Validate weekday range.
- Validate time ranges.
- Validate certificate leave date range.

Indexes:

- Improve lookup speed on role, user profile, slot, appointment, emergency alert, prescription, and certificate queries.

Views:

- Provide reusable report/dashboard summaries.
- Reduce repeated complex joins in application queries.

Triggers:

- Enforce certificate issue date integrity at database level.

Transactions:

- Used for booking, cancellation, completion, report updates, certificate creation, admin role changes, and emergency alert lifecycle changes.

Generated columns:

- appointments.active_slot_id supports double-booking prevention while allowing cancelled slots to be reused.


18. MAIN WORKFLOWS

Authentication workflow:

1. User submits email and password to /auth/login.
2. Auth service checks login attempt lockout.
3. User is loaded by email.
4. Password is verified using bcrypt.
5. JWT token is generated with user_id and role.
6. Frontend stores token and sends it in Authorization header.
7. Protected routes decode token and load current user.

Signup workflow:

1. User submits signup data to /auth/signup.
2. Backend checks email and roll number conflicts.
3. Password is hashed.
4. User row is inserted with default student role.
5. Student profile is inserted.
6. JWT token is returned.

Appointment booking workflow:

1. Patient selects date.
2. Backend lists doctors and slots for that date.
3. Backend generates missing future slots from doctor availability if needed.
4. Patient selects slot.
5. POST /appointments/book is called.
6. Backend locks selected slot.
7. Backend verifies slot is available and not elapsed.
8. Appointment is inserted.
9. Slot status changes to booked.
10. Confirmation response is returned.

Appointment cancellation workflow:

1. User calls PATCH /appointments/{id}/cancel.
2. Backend checks appointment access.
3. Doctor/staff/admin cancellation requires reason.
4. Appointment status changes to cancelled.
5. Slot status changes back to available.
6. Cancellation reason is stored.

Appointment completion workflow:

1. Doctor/admin calls PATCH /appointments/{id}/complete.
2. Backend checks access.
3. Backend rejects cancelled appointments.
4. Appointment status changes to completed.
5. Completed appointments become locked from further medical edits.

Doctor availability workflow:

1. Doctor opens availability page.
2. Backend loads weekly rules and date overrides.
3. Doctor updates weekly availability or date override.
4. Weekly rules affect recurring future slots.
5. Date override affects one date.
6. If doctor becomes unavailable on a date with appointments, those appointments are cancelled automatically.

Medical report workflow:

1. Doctor opens appointment detail.
2. Doctor adds diagnosis and remarks through report notes endpoint.
3. Doctor adds prescription items through prescription endpoint.
4. Patient can view report later through student report endpoints.
5. Completed/cancelled appointments cannot be edited.

Certificate workflow:

1. Doctor/admin issues certificate for appointment.
2. Backend validates appointment context.
3. Backend validates certificate date and leave range.
4. Certificate is inserted or updated.
5. Student can view certificate summary.

Emergency alert workflow:

1. Patient sends emergency alert.
2. Alert stores reason, location, contact number, and message.
3. Admin/staff/doctor can view alerts.
4. Responder acknowledges alert.
5. Responder resolves alert with optional note.
6. Student can view alert status.

Admin role assignment workflow:

1. Admin selects user.
2. Admin assigns role.
3. Backend prevents admin from changing their own role.
4. User role_id is updated.
5. Backend creates or updates the required profile:
   - student/professor/college-staff/hostel-staff use students profile.
   - doctor/staff use staff profile.
6. Result is returned to frontend.

Staff walk-in workflow:

1. Staff searches for existing patient.
2. Staff selects patient and available slot.
3. Staff books appointment as walk-in.
4. Appointment reason is marked as walk-in related.
5. Staff can view walk-in bookings.


19. SECURITY FEATURES

JWT authentication:

- Protected routes require Bearer token.
- Token contains user ID and role.
- Backend reloads user from DB to confirm account still exists and is active.

Role-based access:

- Student/professor/college-staff/hostel-staff: patient workflow.
- Doctor: appointment, patient history, report, certificate, availability workflow.
- Staff: appointment oversight, walk-ins, emergency alert response.
- Admin: full management access.

Password security:

- Passwords are stored as bcrypt hashes.
- Plaintext passwords are never stored.

Rate limiting:

- Fixed-window request limiter is applied globally when enabled.

Login brute-force protection:

- Failed login attempts are tracked.
- Accounts are temporarily blocked after repeated failures.

Idempotency:

- Authenticated write requests require Idempotency-Key.
- Prevents accidental duplicate writes from repeated requests.

Soft deactivation:

- Admin deletion is implemented as users.is_active = false.
- Medical history remains in database.

Parameterized SQL:

- Query modules use parameterized SQL instead of string interpolation.

24. SUMMARY

The backend is organized around a clean layered architecture. FastAPI routes expose endpoints, services enforce business rules, repositories manage transactions, query files contain raw parameterized SQL, and MySQL stores normalized relational data. The system supports authentication, role-based access, appointment booking, doctor availability, student reports, prescriptions, certificates, emergency alerts, staff walk-ins, and admin user management.

The database uses strong DBMS concepts such as primary keys, foreign keys, unique constraints, indexes, check constraints, views, triggers, transactions, and generated columns. The schema is normalized up to 3NF for the MVP and is suitable for generating an ER diagram in MySQL Workbench.
